# TD-003: Generate OpenAPI v4 spec and share with the Portal SPA

## Status

Open. Mirror of Portal `TD-002` ("Replace hand-written `frontend/src/lib/types.ts` with OpenAPI-generated types"). Either side can drive — the practical work needs both.

## Context

When the UI was extracted from this repo into `octopus-components-management-portal` (PR #147, [ADR-012](../adr/012-portal-architecture.md), reversed [ADR-009](../adr/009-ui-repository-strategy.md)), we accepted that API + UI changes are no longer atomic across a single PR. The mitigation we agreed to in ADR-012 §"Negative — and how mitigated" is OpenAPI generation, so the SPA can derive its TypeScript types from a machine-readable contract instead of hand-coding them.

That generation is not yet wired. As of `v3` HEAD:
- The CRS server has no OpenAPI generator dependency in `components-registry-service-server/build.gradle`.
- The Portal SPA hand-maintains `frontend/src/lib/types.ts` against the Kotlin DTOs in this repo.

This is fine while a single team owns both repos and the API surface is small. It will not scale.

## Why this matters

- **Drift is silent.** Hand-written types compile without ever talking to the server; the only signal is a runtime `undefined` somewhere in a form component.
- **Two-PR-across-two-repos chore** for any new field in `ComponentUpdateRequest` etc.
- **Field renames are dangerous** — a rename on this side without a matching rename on the Portal looks like "the field works on read but disappears on save."

## Scope

### CRS side (this repo)

1. Add `springdoc-openapi-starter-webmvc-ui` (or equivalent) to `components-registry-service-server/build.gradle`.
2. Configure springdoc to scan v4 controllers (and `AuthController` at `/auth/me`, which is outside `/rest/api/4`). Exclude legacy v1/v2/v3 — they are stable Feign-client contracts; not worth re-deriving.
3. Add a Gradle task `:components-registry-service-server:generateOpenApiDocs` that emits `build/openapi/v4.json`. Wire it to run as part of `./gradlew build` so a hand-edited spec drift fails CI.
4. Publish `v4.json` as a CI build artifact (TeamCity / GitHub Actions) so the Portal CI can pull it.

### Portal side (mirrors Portal TD-002)

1. Add `openapi-typescript` (or `openapi-fetch`) to dev dependencies.
2. Pull `v4.json` (or vendor a refreshed copy) and generate `frontend/src/lib/api/schema.d.ts`.
3. Replace the hand-written API surface in `frontend/src/lib/types.ts` with imports from the generated schema. Keep a thin SPA-only-types layer (zustand state shape, etc.) that doesn't have a backend counterpart.
4. CI gate: `npm run generate-types -- --check` must fail when the committed schema disagrees with the bundled `v4.json`.

## Out of scope

- Generating Kotlin client code for v4 from the same spec (the existing v1/v2/v3 Feign client is hand-shaped and not part of this work).
- Migrating v1/v2/v3 endpoints to OpenAPI generation. They are the stable legacy contract; only v4 should be auto-generated.
- API versioning beyond v4.

## Acceptance criteria

1. `./gradlew build` produces `build/openapi/v4.json` covering at least:
   - `/rest/api/4/components/**`
   - `/rest/api/4/audit-log/**`
   - `/rest/api/4/admin/**`
   - `/rest/api/4/config/**`
   - `/rest/api/4/info`
   - `/auth/me`
2. The spec is published as a build artifact reachable from Portal CI.
3. CRS CI fails when the committed (or published) spec disagrees with the live regeneration.
4. Portal `frontend/src/lib/api/schema.d.ts` exists and is regenerated by `npm run generate-types`. Hand-written types in `frontend/src/lib/types.ts` no longer duplicate fields that come from a backend DTO.
5. Field-by-field parity between Kotlin DTOs and TypeScript types as of the merge of this work.
